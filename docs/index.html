GymSaaS
Documentación completa

API REST para gestión de gimnasios: autenticación, clientes, membresías, rutinas, asistencia, casilleros, progreso, posts y más. Frontend en Vue 3 + Vite.

ℹ️ Backend: http://localhost:3000 · Frontend: http://localhost:5173

Stack tecnológico

BackendNode.js + Express 5 (ESM)
Base de datosPostgreSQL (postgres.js)
AuthJWT + bcrypt
FrontendVue 3 + Vite 7
EstadoPinia
HTTP clienteAxios

Iniciar el proyecto desde la raíz

En la carpeta raíz del proyecto (donde está el package.json principal):

1. Instalar dependencias (backend y frontend)

# Instala en backend y frontend a la vez
npm run start:project

O por separado:

npm run build:backend   # cd backend && npm install
npm run build:frontend  # cd frontend && npm install

2. Configurar variables de entorno

Crea backend/.env con al menos:

# Conexión PostgreSQL (si usas Docker local → ver paso 3 Opción A; si usas producción → tu URL)
DATABASE_URL=postgresql://usuario:password@host:puerto/nombre_base

# Servidor
PORT=3000

# JWT (string largo aleatorio que puedes definir como quieras, recomendable un uuid)
JWT_SECRET=tu-secreto-superseguro

# Opcional: URL del frontend (para QR de registro)
FRONTEND_URL=http://localhost:5173
⚠️ No subas .env a Git. Debe estar en .gitignore.

3. Crear la base de datos

Puedes usar una base de datos local con Docker (recomendado para desarrollo) o una base en producción (Supabase, etc.).

Opción A — Base de datos local con Docker Compose (recomendado)

Desde la carpeta raíz del proyecto, levanta PostgreSQL con el docker-compose.yml. El contenedor monta la carpeta sql/ como scripts de inicialización, así que sql/init.sql se ejecuta solo la primera vez. No necesitas credenciales de producción.

# En la raíz del proyecto (donde está docker-compose.yml)
npm run docker:init

Esto ejecuta docker compose up -d: Postgres queda en localhost:5432, usuario postgres, contraseña gymsaaspass123, base de datos gymsaas. En backend/.env usa:

DATABASE_URL=postgresql://postgres:gymsaaspass123@localhost:5432/gymsaas

Comandos útiles:

npm run docker:stop    # Detener contenedores
npm run docker:restart # Reiniciar (down + up -d)
✅ Con esta opción no necesitas configurar Supabase ni ninguna DB en la nube para empezar a desarrollar.

Opción B — Base de datos en producción

Si prefieres usar una base de datos ya desplegada (por ejemplo Supabase): crea el proyecto y la base, luego ejecuta manualmente el contenido de sql/init.sql en el SQL Editor del proveedor. En backend/.env configura DATABASE_URL con las credenciales que te proporcione el servicio (usuario, contraseña, host, puerto y nombre de la base).

DATABASE_URL=postgresql://usuario:password@host:puerto/nombre_base

4. Arrancar backend y frontend

# Terminal 1 — Backend
npm run start:backend

# Terminal 2 — Frontend
npm run start:frontend

Backend quedará en http://localhost:3000, frontend en http://localhost:5173.

✅ En la consola del backend deberías ver: Servidor corriendo en puerto 3000 y DB conectada: [...]

Auth — /api/auth

Registro, login y CRUD de usuarios y gyms.

POST Público /api/auth/register

Registro. Tres casos: (1) Cliente sin gym — solo name, email, password. (2) Cliente de un gym existente — añade gym como string (gym_id). (3) Crear gym + admin — gym como objeto { name, address?, phone? }.

Body (JSON)

CampoTipoRequeridoDescripción
namestring✅Nombre del usuario
emailstring✅Email único
passwordstring✅Se hashea con bcrypt
gymstring \| object—UUID del gym (cliente) o { name, address?, phone? } (crear gym + admin)

Ejemplo — Crear gym y admin

{
  "name": "Admin Gym",
  "email": "admin@gym.com",
  "password": "secret123",
  "gym": {
    "name": "Iron Gym",
    "address": "Calle 72 #10-30",
    "phone": "601 456 7890"
  }
}

Respuesta 201

{ "user": { "id", "name", "email", "role", "gym_id" }, "gym": { "id", "name" } }
POST Público /api/auth/login

Login. Body: { "email", "password" }. Devuelve { "token", "user" }. JWT expira en 7 días.

Respuesta 200

{
  "token": "eyJhbGciOiJIUzI1NiIs...",
  "user": {
    "id", "name", "email", "role",
    "gym_id", "gym_name", "client_profile_id"
  }
}
POST 🔒 JWT Admin /api/auth/register-user

Registrar trainer o client desde el dashboard (gym_id del token). Body: name, email, password, role (trainer \| client). Opcionales para client: phone, address, eps, emergency_contact.

PUT 🔒 JWT Admin /api/auth/gym/:id_gym

Actualizar gym. Body: name_gym, address, phone.

DELETE 🔒 JWT Admin /api/auth/gym/:id_gym

Eliminar gym (cascada: usuarios y perfiles).

PUT 🔒 JWT Admin /api/auth/user/:id_user

Actualizar usuario. Body: name, email.

DELETE 🔒 JWT Admin /api/auth/user/:id_user

Eliminar usuario (y perfil en cascada).

Clients — /api/clients

Todos requieren JWT. Admin o trainer: mismo gym. Cliente: solo su propio perfil donde aplique.

GET 🔒 Admin/Trainer /api/clients/:gym_id

Lista de clientes del gym con perfil, membresía activa y días restantes.

GET 🔒 JWT /api/clients/detail/:id_client

Un cliente por ID de perfil (clients_profile.id). Cliente solo puede ver el suyo.

PUT 🔒 Admin /api/clients/detail/:id_client

Actualizar perfil. Body: phone, address, eps, emergency_contact.

DELETE 🔒 Admin /api/clients/:id_user

Eliminar cliente (parámetro es user_id, no id del perfil).

Memberships — /api/memberships

GET 🔒 Admin/Trainer /api/memberships/gym/:gym_id

Lista de planes de membresía del gym.

POST 🔒 Admin /api/memberships

Crear membresía. Body: name, duration_days (número).

POST 🔒 Admin /api/memberships/assign

Asignar membresía a cliente. Body: client_id, membership_id. Si ya tiene activa, se marca como expired y se crea la nueva.

GET 🔒 JWT /api/memberships/client/:client_id

Membresías del cliente (con días restantes). Solo propio cliente o admin/trainer del gym.

PUT 🔒 Admin /api/memberships/:id

Actualizar membresía. Body: name, duration_days.

DELETE 🔒 Admin /api/memberships/:id

Eliminar membresía.

Routines — /api/routines

GET 🔒 Admin/Trainer + gym /api/routines/gym/:gym_id

Rutinas del gym con nombre del trainer.

GET 🔒 JWT /api/routines/my

Mis rutinas (asignadas a mí o creadas por mí si soy cliente/trainer independiente).

GET 🔒 JWT /api/routines/exercises

Catálogo global de ejercicios (para armar rutinas).

POST 🔒 JWT /api/routines

Crear rutina. Body: name (requerido), description, trainer_id, exercises (array de { exercise_id, sets?, reps?, suggested_weight? }).

GET 🔒 JWT /api/routines/:routine_id/exercises

Ejercicios de una rutina (si tienes permiso para verla).

POST 🔒 Admin/Trainer /api/routines/assign

Asignar rutina a cliente. Body: client_id, routine_id.

GET 🔒 JWT /api/routines/client/:client_id

Rutinas asignadas a un cliente (admin/trainer mismo gym o cliente solo el suyo).

POST 🔒 JWT /api/routines/:routine_id/exercises

Añadir ejercicio a rutina. Body: exercise_id (requerido), sets, reps, suggested_weight.

DELETE 🔒 JWT /api/routines/:routine_id/exercises/:exercise_id

Quitar ejercicio de la rutina (exercise_id es el ID en routine_exercises).

Attendance — /api/attendance

POST 🔒 Admin/Trainer /api/attendance/check-in

Registrar entrada. Body: client_id. Requiere gym.

GET 🔒 JWT /api/attendance/my

Mi historial de asistencia (cliente con gym).

GET 🔒 Admin/Trainer /api/attendance/:gym_id?day=YYYY-MM-DD

Historial del gym. Query: day (fecha), week (ej. 2025-W11). Sin query: últimos 30 días.

Lockers — /api/lockers

GET 🔒 JWT + gym /api/lockers/my

Mi casillero asignado (cliente).

GET 🔒 Admin /api/lockers/:gym_id

Lista de casilleros del gym con asignación actual.

POST 🔒 Admin /api/lockers

Crear casillero. Body: locker_number (número).

POST 🔒 Admin /api/lockers/assign

Asignar casillero a cliente. Body: client_id, locker_id.

POST 🔒 Admin /api/lockers/release

Liberar casillero. Body: client_id, locker_id.

DELETE 🔒 Admin /api/lockers/:id

Eliminar casillero (solo si no está ocupado).

Progress — /api/progress

POST 🔒 JWT /api/progress

Registrar progreso. Body: client_id, exercise_id, weight y/o reps. Cliente solo puede su propio client_id.

GET 🔒 JWT /api/progress/client/:client_id?exercise_id=uuid

Progreso de un cliente. Query opcional exercise_id para filtrar.

Posts — /api/posts

GET 🔒 JWT + mismo gym /api/posts/gym/:gym_id

Posts del gym.

GET 🔒 JWT /api/posts/detail/:id

Un post por ID (mismo gym).

POST 🔒 Admin/Trainer /api/posts

Crear post. Body: title (requerido), content, image.

PUT 🔒 Admin/Trainer /api/posts/:id

Actualizar post. Body: title, content, image.

DELETE 🔒 Admin/Trainer /api/posts/:id

Eliminar post.

Exercises — /api/exercises

GET 🔒 JWT /api/exercises

Lista de ejercicios (catálogo global).

POST 🔒 Admin + gym /api/exercises

Crear ejercicio. Body: name (requerido), muscle_group, description.

QR — /api/qr

GET Público /api/qr/register/:gym_id

Datos para generar QR de registro: gym_id, gym_name, register_url (URL del front con ?gym_id=...).

GET 🔒 Admin /api/qr/register/:gym_id/secure

Mismo payload pero con token (solo admin del gym).

Trainers — /api/trainers

Solo admin (gym_id del token).

GET 🔒 Admin /api/trainers

Lista de trainers del gym.

POST 🔒 Admin /api/trainers

Registrar trainer. Body: name, email, password.

PUT 🔒 Admin /api/trainers/:id

Actualizar trainer. Body: name, email.

DELETE 🔒 Admin /api/trainers/:id

Eliminar trainer.

Autenticación con JWT

Endpoints protegidos requieren el header:

Authorization: Bearer <token>

El token se obtiene con POST /api/auth/login. Contiene id, role, gym_id, client_profile_id. Expira en 7 días.

CódigoMotivo
401Falta token o token inválido/expirado
403Rol insuficiente o recurso de otro gym

Frontend (Vue 3 + Vite)

El frontend usa VITE_API_URL (por defecto http://localhost:3000/api).

Servicio API

frontend/src/services/api.js: instancia de Axios con baseURL, interceptor de request que añade Authorization: Bearer <token> desde localStorage, e interceptor de response que ante 401 limpia token y redirige a /login.

// Uso típico en un componente
import api from '@/services/api.js'

// GET
const { data } = await api.get('/clients/' + gymId)
// POST
const { data } = await api.post('/auth/register', body)

Rutas y roles

Rutas principales en router/routes.js:

RutaVistaRequisitos
/login, /registerLogin, RegisterInvitado (si ya hay sesión → dashboard)
/dashboardDashboardAuth
/clientsClientPageAuth, gym, admin o trainer
/trainersTrainersViewAuth, gym, admin o trainer
/membershipsMembershipsViewAuth, gym, admin
/routinesRoutinesViewAuth
/attendanceAttendanceViewAuth, gym
/lockersLockersViewAuth, gym
/progressProgressViewAuth
/postsPostsViewAuth, gym
/qrQrViewAuth, gym, admin

El store auth.store.js guarda usuario y token; el router comprueba requiresAuth, requiresGym y roles antes de entrar a cada ruta.

Ejemplos de uso de la API

1. Login (fetch)

const res = await fetch('http://localhost:3000/api/auth/login', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({ email: 'admin@gym.com', password: 'secret123' })
})
const data = await res.json()
// Guardar token para siguientes peticiones
localStorage.setItem('token', data.token)

2. Registrar gym + admin (público)

const res = await fetch('http://localhost:3000/api/auth/register', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    name: 'Admin',
    email: 'admin@gym.com',
    password: 'pass123',
    gym: { name: 'Iron Gym', address: 'Calle 1', phone: '600 111 2222' }
  })
})
const { user, gym } = await res.json()

3. Obtener clientes del gym (con token)

const token = localStorage.getItem('token')
const res = await fetch(`http://localhost:3000/api/clients/${gymId}`, {
  headers: { 'Authorization': `Bearer ${token}` }
})
const clientes = await res.json()

4. Asignar membresía a cliente

await fetch('http://localhost:3000/api/memberships/assign', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({ client_id: clientId, membership_id: membershipId })
})

5. Check-in de asistencia

await fetch('http://localhost:3000/api/attendance/check-in', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({ client_id: clientId })
})

6. Crear rutina con ejercicios

const res = await fetch('http://localhost:3000/api/routines', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({
    name: 'Rutina fuerza',
    description: '3 días por semana',
    exercises: [
      { exercise_id: exerciseUuid1, sets: 3, reps: 10, suggested_weight: 60 },
      { exercise_id: exerciseUuid2, sets: 3, reps: 12 }
    ]
  })
})
const rutina = await res.json()

7. Registrar progreso

await fetch('http://localhost:3000/api/progress', {
  method: 'POST',
  headers: {
    'Content-Type': 'application/json',
    'Authorization': `Bearer ${token}`
  },
  body: JSON.stringify({
    client_id: clientId,
    exercise_id: exerciseId,
    weight: 70,
    reps: 8
  })
})

8. Obtener datos para QR de registro (público)

const res = await fetch(`http://localhost:3000/api/qr/register/${gymId}`)
const { gym_id, gym_name, register_url } = await res.json()
// register_url = frontend + ?gym_id=...

Códigos de error

CódigoSignificadoCuándo
400Bad RequestCampos requeridos faltantes o formato inválido
401UnauthorizedSin token o token inválido/expirado
403ForbiddenRol insuficiente o recurso de otro gym
404Not FoundRecurso no existe
409ConflictEmail/gym duplicado, casillero ocupado, etc.
500Server ErrorError interno (mensaje en error o message)

Las respuestas de error suelen ser JSON con message o error.
